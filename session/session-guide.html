Section 2: OTel GenAI Semantic Conventions

Deep dive into the three span types for agentic observability:

  • invoke_agent — agent.name, agent.id, agent.version, provider.name
  • chat — request.model, response.model, usage.input/output_tokens
  • execute_tool — tool.name, tool.call.id, arguments, result (opt-in)

Show the span hierarchy: invoke_agent → chat (plan) → execute_tool* → chat (synthesize)

Speaker Notes

Before the demo: Explain the 3 span types on the slide. "Think of invoke_agent as the envelope, chat as the LLM thinking, and execute_tool as the hands doing the work."

These are not custom — they're part of the OTel GenAI semantic conventions spec. Any OTel-compatible backend renders them.

40:00 — 10 min

Section 4: MCP Tracing — The Cross-Process Bridge

  • MCP = USB for AI agents (universal tool protocol)
  • Problem: each MCP call is a new process/context boundary
  • Solution: W3C traceparent in params._meta
  • OTel MCP semantic conventions: mcp.method.name, mcp.protocol.version, mcp.session.id
  • Span pairs: CLIENT span (agent) + SERVER span (MCP server)
Speaker Notes

Context-set: "MCP is becoming the USB for AI agents — a universal tool protocol. Copilot uses it, Claude uses it, your custom agents will use it. But without trace propagation, each MCP call is a black box."

Show the JSON-RPC structure. "One line in _meta — that's all it takes to stitch together a distributed trace across MCP boundaries."

Section 5: Field Patterns & Anti-Patterns

From the field (Cognizant, Trizetto, Infosys):

  • Cognizant: 100+ apps migrated AppDynamics → App Insights for agent workloads on AKS
  • Trizetto: Per-tenant trace isolation for HIPAA compliance, Semantic Kernel agents
  • Infosys: 150+ tools, multi-agent orchestration, token cost attribution across BUs

Anti-patterns:

  • ❌ Logging PII in traces — use opt-in content capture (OTEL_INSTRUMENTATION_GENAI_CAPTURE_MESSAGE_CONTENT)
  • ❌ Missing trace context across service boundaries — always propagate traceparent
  • ❌ 100% sampling in production — use TraceIdRatioBased, start at 10%
  • ❌ Custom-only attributes — use standard GenAI semconv first, then extend
  • ❌ Traces without dashboards — data without visibility is just storage cost
Speaker Notes

Share specific stories:

"Cognizant — their LangChain agents were producing great results, but the ops team had zero visibility. We added opentelemetry-instrumentation-openai-v2, routed to App Insights, and within a week they could see token consumption per agent per customer."

"Trizetto — healthcare, HIPAA. Every data access must be auditable. We added governance.data.sensitivity attributes and data.access.audit events — exactly what you saw in Demo 3. Their compliance officer signed off in one review."

"Infosys — 150+ tools, Copilot Studio's 15-tool limit was a wall. They built a custom orchestrator. Challenge: cost attribution across 8 business units. Token counts on agent spans, budget codes as attributes — problem solved."

Anti-patterns — quick hits:

"Number one mistake I see: logging everything including PII. Default content capture is OFF for a reason. Opt in only for debug envs."

"Number two: 100% sampling in production. We saw one team capturing every span for 200 agents — their App Insights bill was $4K/month. Sampling at 10% dropped it to $400 with no loss of insight."

Q&A prep — likely questions:

"What about PII?" → Default content capture is OFF. Set OTEL_INSTRUMENTATION_GENAI_CAPTURE_MESSAGE_CONTENT=true only in dev. Add PII scrubbing span processors for production.

"What about sampling?" → TraceIdRatioBased sampler. 10% in production, 100% in dev. App Insights also has adaptive sampling.

"Can I use Jaeger/Datadog instead?" → Absolutely. OTel is backend-agnostic. Add any OTLP-compatible exporter.

"Streaming responses?" → Events capture chunks. Total tokens counted at stream completion.

"Multi-agent correlation?" → Trace ID propagates. Each agent gets an invoke_agent span under the same trace.

"App Insights cost?" → Log Analytics pricing, ~$2-5/GB. Use sampling to control volume.
